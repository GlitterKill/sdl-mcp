# LadybugDB 0.18.1 Windows FTS Runtime Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents are available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

> **Revision:** `3.5.7-sdl.1` is retired because its published provenance named the wrong OpenSSL release signer. The exact supported and registry-verified runtime boundary is `3.5.7-sdl.2`; ranges and `.1` compatibility are intentionally unsupported.

**Goal:** Upgrade SDL-MCP to official `@ladybugdb/core@0.18.1` without reintroducing the Windows native-process crash, while temporarily supplying `libssl-3-x64.dll` and `libcrypto-3-x64.dll` from a deterministic SDL-owned Windows package.

**Architecture:** Keep Ladybug itself official and unmodified through the existing `kuzu` npm alias. Publish one Windows/x64-only data package containing exactly the two OpenSSL runtime DLLs, their license, and machine-verifiable provenance. Reuse SDL's existing Rust napi addon to load the verified package DLLs by absolute path—`libcrypto` first, then `libssl`—with `LoadLibraryExW(..., LOAD_LIBRARY_SEARCH_DLL_LOAD_DIR | LOAD_LIBRARY_SEARCH_DEFAULT_DIRS)` immediately before Ladybug's `LOAD EXTENSION fts`. Keep the two preload handles until Ladybug returns, then release them in reverse order in `finally`. Ladybug's own FTS module reference then owns its dependency references. This avoids global `PATH`/`SetDllDirectory` changes and avoids writing into Ladybug's extension cache. Missing packages, missing/incompatible native shims, or preload failures disable only FTS through the existing per-extension capability boundary.

**Tech Stack:** TypeScript/ESM, Node.js 24, Rust/napi-rs, Windows `LoadLibraryExW`, `node:test`, LadybugDB 0.18.1, OpenSSL 3.5.7 LTS built from official source with MSVC, npm platform packages, PowerShell, GitHub Actions Windows runners.

**Design baseline:** This plan extends and then retires the workaround described in `devdocs/implemented/specs/2026-07-13-live-index-symbol-fts-crash-fix-design.md`. The upstream facts are pinned to LadybugDB v0.18.1: core links OpenSSL 3, the Windows loader calls plain `LoadLibraryW` on the FTS extension, the installer stores it under `%USERPROFILE%/.lbdb/extension/0.18.1/win_amd64/fts`, and the extension collector omits the two OpenSSL DLLs. Microsoft documents that dependencies of a full-path load are still searched by module name; the plugin directory is searched only with an altered/load-directory strategy. Therefore copy-only side-by-side placement is explicitly not the fix.

**Expert verification:** The Ladybug DB expert rejected the copy-only draft and approved this corrected boundary: official 0.18.1 core, OpenSSL 3.5.7/`VC-WIN64A shared`, a data-only Windows package paired with the existing SDL native addon, scoped preload-handle lifetime inside `loadExtensionsOnConnection()`, FTS-only degradation, mandatory real-patch regression, and non-downgrade rollback.

## Decision and tradeoffs

### Selected: data-only Windows runtime package plus scoped existing-addon preload

Use `@sdl-mcp/ladybug-openssl-win32-x64@3.5.7-sdl.2` as an exact optional dependency of `sdl-mcp`, and add two internal Windows-loader functions to the existing `sdl-mcp-native` addon.

**Pros**

- Keeps `@ladybugdb/core` official and makes the workaround deletable.
- npm's `os`/`cpu` filters prevent non-Windows downloads.
- The OpenSSL artifact can be rebuilt/audited independently of SDL releases.
- Exact pinning makes installs deterministic and security rollouts explicit.
- Absolute-path `LoadLibraryExW` preloads use the package directory without process-global DLL search changes.
- Direct dependency preload avoids writes/races/foreign-file conflicts in `%USERPROFILE%/.lbdb`.

**Cons**

- Windows FTS temporarily requires both optional packages: the OpenSSL data package and an SDL native addon version containing the loader shim.
- SDL owns OpenSSL patch monitoring, build integrity, npm publication, incident response, license/provenance data, and native shim maintenance until upstream fixes both packaging and loading.
- A security update requires publishing a new runtime package and an SDL patch release that updates the exact optional-dependency pin.
- Users who omit/disable optional native dependencies retain the server but lose FTS-backed capability.

### Rejected: copy the DLLs beside `libfts.lbug_extension` without a loader change

Ladybug 0.18.1 calls plain `LoadLibraryW`. Windows resolves dependent DLLs by module name using the standard process search order, which does not include the plugin directory. Copy-only placement looks correct but still fails on a clean machine.

### Rejected: mutate `PATH`, call `SetDllDirectory`, or use `AddDllDirectory` process-wide

These approaches change process-global loader behavior and introduce races with unrelated native loads. A scoped absolute-path preload is narrower and easier to prove.

### Rejected: revive `@sdl-mcp/ladybug` and republish Ladybug

This expands SDL's responsibility to the Ladybug binary and conflicts with the explicit goal of using official `@ladybugdb/core@0.18.1`.

### Rejected: put both DLLs in the main `sdl-mcp` tarball

This makes every platform download Windows binaries and couples every OpenSSL rebuild to the main tarball.

### Rejected: use Git, Conda, or machine-installed OpenSSL DLLs

This is not reproducible and creates an uncontrolled security/provenance boundary. It is diagnostic evidence only.

## Stop conditions and rollback

Stop the rollout and keep 0.16.1 plus `withSymbolFtsPausedForPatch()` if any of these occurs:

- OpenSSL source signature/checksum cannot be verified.
- Staged exports do not satisfy every official FTS OpenSSL import.
- DLL imports include an unapproved non-system runtime.
- Scoped native preload does not produce package-root module origins.
- `LOAD EXTENSION fts` still fails in the sanitized environment.
- Any child exits with `0xC0000005` or hangs.
- Normal startup fails when optional dependencies are omitted/disabled.
- The registry tarball differs from the verified CI artifact.
- The v40→v42 migration fixture fails or Ladybug 0.18.1 introduces an unrelated compatibility regression.

Rollback is not an ordinary dependency downgrade. Ladybug 0.18.1 can checkpoint/upgrade v40/v41 storage to v42, and 0.16.1 cannot reopen that upgraded database.

Before first 0.18.1 open, require a verified v40 backup or a tested export/restore of every non-derived table. A backup is verified only after restoring it to a separate path and opening it successfully with 0.16.1. Rebuild-only rollback is permitted only when an archived preflight query proves the specific database has zero non-derived memory, feedback, usage, and audit rows. If rollback is required:

1. stop all SDL processes;
2. preserve the failed v42 database for diagnosis;
3. restore the verified v40 backup, or restore every non-derived table from the tested export and rebuild derived graph state under 0.16.1; use rebuild-only recovery only with the recorded zero-non-derived-state proof;
4. restore the 0.16.1 dependency and `withSymbolFtsPausedForPatch()`;
5. rerun health/index/FTS checks before reopening normal writes.

If SDL OpenSSL/native packages were already published, deprecate affected npm versions with recovery guidance; do not unpublish immutable provenance evidence.
