Go adapter's `extractCalls` built call ranges with `startPosition.row` (0-based tree-sitter) while every other adapter and the core treesitter extraction uses `startPosition.row + 1` (1-based). Rust adapter had the same bug in three places: macro calls, regular calls, and chained calls. This caused incorrect caller attribution via `findEnclosingSymbol` which compares 1-based symbol ranges against call ranges. Fixed by adding `+ 1` to all row values in call range construction. The Go adapter's `extractRange` helper (used for symbols) already did `+ 1` correctly — only the inline call range construction was wrong.
